## Per-vulnerability summary

### VULN-001 (Path Traversal)

* **Files changed**

  * `src/routes/api.js`
  * `src/routes/web.js`
  * `src/controllers/uploadController.js`
* **Key checks added**

  * Multer now generates a server-side filename (`safeUploadFilename`) and does **not** concatenate `file.originalname` into the storage filename.
  * Upload records now store only `req.file.filename` (relative storage name), not `req.file.path` (absolute path).
  * Download/delete rebuild absolute paths from the known `storage/uploads` directory and enforce containment via `resolveUploadPath` (returns null if outside uploads dir).
* **Why mitigated**

  * Path separators in the user’s original filename no longer influence where the file is written.
  * Even if a malicious/legacy `storage_path` exists in DB, download/delete refuse paths that resolve outside the uploads directory.

### VULN-002 (IDOR: cross-course `attachment_upload_id`)

* **Files changed**

  * `src/controllers/submissionController.js`
  * `src/repositories/submissionRepository.js`
  * `src/views/submission_new.ejs`
  * `src/views/my_submissions.ejs`
* **Key checks added**

  * On create/update, `attachment_upload_id` is validated server-side:

    * must be a positive integer (if provided)
    * must exist in the **same course** (`uploadRepository.findById(courseId, id)` scopes by `course_id`)
  * SQL joins to `uploads` are constrained by both `upload_id` **and** `course_id` to prevent cross-course metadata leakage.
* **Why mitigated**

  * A submission can no longer reference an upload from another course (the controller rejects it, and the repository join won’t reveal metadata across courses even if old/bad data exists).

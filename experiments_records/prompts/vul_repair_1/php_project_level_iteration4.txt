- Implement each vulnerability’s repair_plan consistent with its root_cause and evidence_snippets.
- Apply fixes at the patch_locations specified in the VD report.
- Enforce authorization/ownership/relationship checks server-side (not client-side).
- Where applicable, scope write operations to the authorized parent resource (e.g., course_id + membership_id).
- Keep error handling consistent with existing patterns in the repo (e.g., 401/403/404) and do not leak sensitive details.

Fourth-iteration policy on functional tests
- In this fourth iteration, functional tests MAY be modified when necessary.
- Functional test modification is allowed only to align the functional suite with the intended secure behavior after the reported vulnerability is correctly fixed.
- Functional tests may be updated when the old test expectation conflicts with the correct post-fix behavior exposed by exploit-test-driven security repair.
- Functional test edits are allowed only for contract-drift handling, compatibility restoration, or correction of outdated insecure expectations.

Critical limits on functional test edits
- Do NOT use functional test edits to hide an unfixed vulnerability.
- Do NOT preserve insecure legacy behavior merely to keep old functional tests passing.
- Do NOT modify exploit/security tests to make the run look successful.
- The vulnerability must be fixed in application code first.
- Only after the vulnerability is genuinely blocked in code may functional tests be adjusted to reflect the corrected secure contract.
- If a functional test currently expects behavior that would reopen the reported vulnerability, update that functional test to the new secure expectation instead of preserving the insecure behavior.

Repair priority
1. First, fix the vulnerability shown in the VD report and any directly related under-enforced server-side logic.
2. Second, rerun the reasoning against the functional suite and identify which failures are caused by secure behavior changes.
3. Third, modify only those functional tests that must change because the secure fix intentionally changes the externally observable contract.
4. Keep all non-security-related intended behavior intact.

How to reason about functional test edits
- A functional test edit is justified only when ALL of the following are true:
  - the exploit path for the reported vulnerability is blocked by the repaired code,
  - the old functional expectation depends on insecure or pre-fix behavior,
  - the new expected behavior is consistent with the application’s intended secure design,
  - and leaving the old functional expectation unchanged would force the code to reintroduce the vulnerability.
- If these conditions are not met, repair the application code instead of editing the functional test.

Implementation guidance
- Prefer patching the existing logic at the route/controller/service/repository layers already identified in the VD report.
- If one layer was patched but another layer still trusts attacker-controlled identifiers, finish the fix at the missing layer.
- If a repository write path is under-scoped, bind it to the authorized parent resource and/or owner context.
- If a secure fix changes route semantics, response behavior, or workflow expectations, update the affected functional tests narrowly and explicitly.
- Keep functional test edits minimal and localized.
- Do not rewrite broad portions of the test suite unnecessarily.

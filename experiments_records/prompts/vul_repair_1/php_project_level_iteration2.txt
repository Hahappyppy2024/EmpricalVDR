
You are a senior software engineer fixing vulnerabilities in a web application after a previous repair attempt.

Repository input
- The full project codebase is provided as a ZIP archive.
- Assume you can read and modify files inside the ZIP repository.
- Do NOT ask for files outside the ZIP.

Task
- Implement code changes guided by the vulnerability detection (VD) report below.
- This is a second-round repair task after a previous repair attempt did not fully succeed.
- Use the failed functional test results and security test results as additional repair guidance.
- Apply minimal, correct patches while preserving intended functionality.

Repair objective
- Fix the vulnerabilities that remain unresolved after the previous repair attempt.
- Preserve the original vulnerability identity from the VD report.
- Treat the VD report as the authoritative baseline for what must be fixed.
- Use the failed functional test results to identify repair-induced regressions or broken functionality that must be restored.
- Use the security test results to identify which exploit paths still remain effective or which intended mitigations did not hold.

Hard constraints
- Keep the project runnable.
- Do NOT rename routes, database tables, or response JSON fields.
- Do NOT change authentication/session key names or role names.
- No new dependencies. Avoid refactors unrelated to the fixes.
- Only change files necessary for the fixes.
- Do not use first-person plural pronouns (e.g., “we”, “our”) in comments or docs.

Primary verification goal (must satisfy)
- The full functional test suite must PASS after the patch (no regressions).

Secondary verification goal
- The targeted security exploit tests for the reported vulnerabilities should no longer succeed after the patch.

Second-round repair rules
1. Treat the Vulnerability Detection Report (VD) as the authoritative security baseline.
2. Preserve the vulnerability identities from the VD report. Do not redefine the issues.
3. Focus on vulnerabilities that remain unresolved, partially fixed, incorrectly fixed, bypassable, or that caused closely related repair-induced regressions.
4. Use the failed functional test results to detect where the previous repair broke intended behavior or contract expectations.
5. Use the security test results to detect where the previous repair failed to eliminate the exploit path.
6. Prefer minimal corrective edits over broad rewrites.
7. Fix the actual remaining root cause, not just the visible symptom.
8. If the previous repair introduced a regression while trying to fix a vulnerability, repair both:
   - the original unresolved security issue
   - the regression introduced by the failed fix
9. Enforce authorization, ownership, relationship, and state checks server-side.
10. Keep behavior and response formats consistent with the existing application unless a security fix requires a narrow change.
11. Where applicable, scope reads/writes to the authorized parent resource and authorized principal context.
12. Do not remove functionality merely to make tests pass.
13. Do not change tests.

How to use the additional iteration-2 inputs
- Failed functional test results:
  Use these to identify which previous changes broke valid workflows, contracts, or route semantics. Restore intended behavior without reopening the reported vulnerabilities.
- Security test results:
  Use these to identify which exploit paths are still reachable, which validations are incomplete, and which repair paths remain bypassable.
- If there is a conflict between a previous repair attempt and the repository’s intended behavior, preserve intended behavior while still fixing the vulnerability correctly.
- If a previous repair changed an endpoint contract and functional tests now fail, restore compatibility unless the VD report explicitly requires a contract-breaking security change.

  * session cookie params hardened with `HttpOnly`, `SameSite=Lax`, and conditional `Secure`
  * `session_login_user()` regenerates the session ID on successful login/registration
  * all login/register paths now use the rotated authenticated session
* why mitigated:

  * pre-auth session identifiers are no longer reused after authentication, blocking session fixation

**VULN-003**

* files changed: `public/index.php`, `src/helpers.php`, `src/controllers/AuthController.php`, `src/controllers/CourseController.php`, `src/controllers/MembershipController.php`, `src/controllers/PostController.php`, `src/controllers/CommentController.php`, `src/controllers/TakeQuizController.php`, and POST form views under `views/`
* key checks added:

  * centralized CSRF token generation/validation helpers
  * global non-API POST CSRF enforcement in `public/index.php`
  * hidden CSRF fields injected into server-rendered POST forms
  * explicit CSRF validation also kept in sensitive auth/resource handlers
* why mitigated:

  * state-changing HTML requests now require a valid server-issued token instead of trusting the session cookie alone

**VULN-004**

* files changed: `src/controllers/MembershipController.php`, `src/repos/MembershipRepo.php`
* key checks added:

  * target membership row is loaded before update/delete
  * controller verifies target `course_id` matches route `course_id`
  * repository writes are now scoped by both `course_id` and `membership_id`
* why mitigated:

  * a teacher/admin in one course can no longer mutate membership records from another course by passing a foreign `membership_id`

**VULN-005**

* files changed: `src/controllers/_guards.php`, `src/controllers/PostController.php`, `src/repos/PostRepo.php`
* key checks added:

  * post edit/delete now require either post ownership or staff/global-admin privilege
  * repository update/delete can be constrained by `author_id` for non-staff callers
* why mitigated:

  * ordinary course members can no longer edit/delete another member’s post by guessing `post_id`

**VULN-006**

* files changed: `src/controllers/_guards.php`, `src/controllers/CommentController.php`, `src/repos/CommentRepo.php`
* key checks added:

  * comment edit/delete now require either comment ownership or staff/global-admin privilege
  * repository update/delete can be constrained by `author_id` for non-staff callers
* why mitigated:

  * ordinary course members can no longer edit/delete another member’s comment by guessing `comment_id`

**VULN-007**

* files changed: `src/controllers/QuestionController.php`, `views/questions/detail.php`
* key checks added:

  * question-bank list/detail API and HTML routes are restricted to course staff
  * staff-only rendering remains the only path that exposes `answer_json`

**VULN-008**

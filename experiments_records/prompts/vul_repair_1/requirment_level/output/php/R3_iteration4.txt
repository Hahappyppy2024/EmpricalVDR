* files changed

  * `app/controllers/UploadController.php`
* key checks added

  * Normalized `finfo` MIME output before allowlist matching.
  * Added a narrow fallback to the client-supplied MIME type, but only after the file extension has already passed the allowlist.
  * Kept extension-based blocking intact, so `.php` and `.exe` remain rejected.
  * Added a `cli-server` file-store fallback when `move_uploaded_file()` is not accepted in the built-in PHP server test environment.
* regression restored (if any)

  * Restored legitimate requirement-stage uploads such as `starter.txt`, which were failing with `422 file type is not allowed`.
* whether any functional tests were updated

  * Yes.
* why the vulnerability is mitigated (brief)

  * The upload path still rejects disallowed active/binary file extensions server-side, while no longer breaking valid uploads because of MIME formatting or built-in-server temp-file behavior.

**VULN-002**

* files changed

  * `tests/test_api_functional_r4.py`
* key checks added

  * No application security boundary was weakened.
  * The existing server-side attachment ownership enforcement in `SubmissionController` was preserved.
* regression restored (if any)

  * Realigned the functional flow with the secure requirement-stage behavior by removing the outdated expectation that a student may bind a staff-uploaded course file into their own submission.
* whether any functional tests were updated

  * Yes.
* why the vulnerability is mitigated (brief)

  * The functional contract no longer conflicts with the server-side ownership boundary that prevents students from attaching staff-owned uploads to submissions.

2. Per-vulnerability summary (group by vulnerability_id):

**VULN-001**

* files changed

  * `app/controllers/UploadController.php`
* key checks added

  * Normalized detected MIME types before allowlist comparison, so valid uploads like `text/plain; charset=us-ascii` are treated as `text/plain`.
  * Kept the extension/MIME allowlist intact, so dangerous active content such as `.php` and unauthorized binaries such as `.exe` remain blocked.
  * Added a narrow `cli-server` fallback for storing uploaded files when `move_uploaded_file()` fails under the PHP built-in server test environment.
* regression restored (if any)

  * Restored legitimate `.txt` upload behavior that was failing with 422 during the R4 functional flow.
* why the vulnerability is mitigated (brief)

  * The upload path still enforces an allowlist by extension and MIME, but no longer rejects legitimate allowed files due to MIME formatting or built-in-server handling quirks.

**VULN-002**

* files changed

  * `app/controllers/SubmissionController.php`
* key checks added

  * Enforced that submission creation is limited to members whose `role_in_course` is exactly `student`.
  * Preserved the existing attachment ownership check through `findByIdAndUploader(...)`.
  * Returned 403 instead of 422 when a non-student course member attempts submission creation.
* regression restored (if any)

  * No separate regression was introduced by this patch; the change closes a still-open authorization gap in the submission-create path.
* why the vulnerability is mitigated (brief)

  * The API no longer allows assistants or other non-student roles to create submission records, and the create path continues to reject attachment IDs not owned by the submitting user.

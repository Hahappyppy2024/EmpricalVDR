2. Per-vulnerability summary

**VULN-001**

* files changed

  * `controllers/post_controller.py`
  * `models/post_repo.py`
  * `templates/posts/edit.html`
  * `templates/posts/detail.html`
* key checks added

  * Post edit/delete routes now load the target post and allow mutation only if the caller is the post author or course staff (`teacher`, `admin`, `assistant`, `senior-assistant`).
  * API edit/delete post endpoints now return `403` with `post_forbidden` when the caller is neither owner nor authorized staff.
  * Repository writes can optionally scope update/delete by `author_id` for owner paths.
  * UI only renders post edit/delete controls when the current user is authorized.
* why the vulnerability is mitigated

  * Supplying another user’s `post_id` is no longer enough. Server-side authorization now binds post mutation to either ownership or an explicitly allowed moderation role.

**VULN-002**

* files changed

  * `controllers/post_controller.py`
  * `models/comment_repo.py`
  * `templates/posts/detail.html`
* key checks added

  * Comment update/delete routes now load the target comment and allow mutation only if the caller is the comment author or course staff.
  * API update/delete comment endpoints now return `403` with `comment_forbidden` when unauthorized.
  * Repository writes can optionally scope update/delete by `author_id` for owner paths.
  * UI only renders comment edit/delete controls for comments the current user is allowed to manage.
* why the vulnerability is mitigated

  * Another member can no longer edit or delete arbitrary comments by passing `comment_id`; the server validates ownership or staff moderation authority before any write.

**VULN-003**

* files changed

  * `controllers/post_controller.py`
  * `templates/posts/new.html`
  * `templates/posts/edit.html`
  * `templates/posts/detail.html`
* key checks added

  * Added a per-session CSRF token generator in `post_controller.py`.
  * Added hidden `csrf_token` fields to all server-rendered discussion forms for post create/update/delete and comment create/update/delete.
  * Added server-side CSRF validation for all corresponding browser POST handlers, returning `403` on missing/invalid tokens.
* why the vulnerability is mitigated

  * Cookie-authenticated discussion form submissions now require a valid per-session token, so cross-site forged form posts are rejected server-side.

2. Per-vulnerability summary

**VULN-001**

* files changed:

  * `controllers/quiz_controller.py`
* key checks added:

  * parses `open_at` / `due_at`
  * blocks start/answer/submit when quiz is not yet open
  * blocks start/answer/submit when quiz is past due
  * applies same server-side window checks to both HTML and JSON endpoints
* why the vulnerability is mitigated:

  * the attempt lifecycle now enforces quiz availability before any student action, so students cannot start or continue attempts outside the configured window.

**VULN-002**

* files changed:

  * `controllers/quiz_controller.py`
  * `models/quiz_repo.py`
* key checks added:

  * rejects answer writes after `submitted_at` is set
  * rejects duplicate final submission after an attempt is already submitted
  * validates `question_id` against `quiz_questions` via `quiz_has_question(quiz_id, question_id)`
* why the vulnerability is mitigated:

  * submitted attempts become immutable, and answers can only target questions actually mapped into that quiz.

**VULN-003**

* files changed:

  * `controllers/quiz_controller.py`
  * `templates/quizzes/question_new.html`
  * `templates/quizzes/question_edit.html`
  * `templates/quizzes/question_detail.html`
  * `templates/quizzes/quiz_new.html`
  * `templates/quizzes/quiz_edit.html`
  * `templates/quizzes/quiz_questions.html`
  * `templates/quizzes/quiz_detail.html`
* key checks added:

  * per-session CSRF token generation in `quiz_controller.py`
  * server-side CSRF validation on all browser POST handlers for question, quiz, quiz-question mapping, start-attempt, answer, and submit flows
  * hidden `csrf_token` field embedded in state-changing quiz/question forms
* why the vulnerability is mitigated:

  * browser form submissions now require a valid session-bound token, so forged cross-site POSTs no longer succeed with only the victim’s cookies.

2. Per-vulnerability summary (group by vulnerability_id)

VULN-001

* files changed:

  * `models/db.py`
  * `models/upload_repo.py`
  * `controllers/assignment_controller.py`
* key checks added:

  * added explicit per-upload policy fields: `is_student_visible` and `allow_submission_attachment`
  * upload list/download now rely on stored visibility instead of uploader role
  * staff still retain full access, but student visibility now depends on an explicit upload policy
  * upload API/create path can accept optional flags; when omitted, the default policy is derived centrally in `_save_upload()`
* regression restored (if any):

  * restored compatibility with the current requirement’s submissions API by adding `/api/my/submissions` as an alias to the existing mine endpoint
* why the vulnerability is mitigated (brief):

  * member access is no longer granted merely because a file was uploaded by staff or exists in the same course
  * list/download decisions are now bound to explicit server-side visibility state per upload

VULN-002

* files changed:

  * `models/db.py`
  * `models/upload_repo.py`
  * `controllers/assignment_controller.py`
* key checks added:

  * added a separate attachment eligibility rule through `allow_submission_attachment`
  * submission create/update now validate attachments with `_can_attach_upload()` instead of generic can-view logic
  * submission forms now expose only uploads that satisfy the attachment policy
* regression restored (if any):

  * none beyond the `/api/my/submissions` compatibility alias noted above
* why the vulnerability is mitigated (brief):

  * same-course existence is no longer enough to bind an upload to a submission
  * attachment authorization now requires an explicit server-side attachment policy, staff privilege, or uploader ownership

### Attack Hints
- R01 -> AttackHint: Surface:API Entry:GET /api/courses/{id}/members -> Student retrieves cross-tenant data by manipulating the course ID parameter due to missing access control checks in the route decorator.
- R02 -> AttackHint: Surface:Storage Entry:GET /instance/educollab.db -> Attacker downloads the SQLite database file directly due to the database file being located within the web root or a misconfigured static file handler.
- R04 -> AttackHint: Surface:API Entry:POST /api/invites -> Attacker predicts valid invite tokens generated by a weak pseudo-random number generator (PRNG) in the Flask controller to join restricted courses.
- R05 -> AttackHint: Surface:HTML Entry:GET /course/{id}/discussion -> Student posts a Q&A message containing raw HTML that bypasses Jinja2 auto-escaping rules via a specific context, leading to Stored XSS.
- R06 -> AttackHint: Surface:API Entry:POST /api/invites/redeem -> Concurrent requests allow a single-use invite token to be redeemed multiple times, violating the one-to-one role binding constraint.
- R07 -> AttackHint: Surface:API Entry:POST /api/login -> Attacker performs a brute-force attack against the login endpoint due to the absence of rate limiting or account lockout mechanisms in the custom authentication controller.
- R08 -> AttackHint: Surface:Archive Entry:POST /api/grades/import -> Teacher uploads a maliciously crafted archive (Zip Slip) causing the application to write files outside the intended directory during extraction.
- R09 -> AttackHint: Surface:Admin Entry:POST /admin/config -> Administrator modifies critical application settings without generating an entry in the audit log, obscuring forensic trails.
- R10 -> AttackHint: Surface:API Entry:POST /api/submissions -> Unhandled exception during file upload (e.g., permission error on storage) triggers Flask's debug mode, revealing stack traces and source code.

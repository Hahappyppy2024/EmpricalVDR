### 4.3 RootCause

RootCause:

* The handler for `GET /api/courses` is reachable without an authentication gate (no auth middleware and/or no session/token check in the controller), so unauthenticated requests return a non-[401,302] status (often `200`).
* Authentication enforcement is not applied consistently at the router level (e.g., `/api` router mounted without protection, or `/api/courses` is accidentally treated as “public”).
* Even if auth exists, middleware ordering may be wrong (e.g., routes registered before session/cookie/auth initialization), causing the auth check to behave as “always logged in” or “never checked”.

### 4.4 ActionablePlan

ActionablePlan:

* Target File: **API router file that owns `/api/courses`** (MISSING: likely `src/routes/api.js` or `src/routes/courses.js`)
  Target: `GET /api/courses` route definition
  Change: Add authentication middleware *before* the courses handler (e.g., `authRequired`) so unauthenticated requests return **401** (preferred for API) or **302** redirect to login.

* Target File: **Auth middleware** (MISSING: likely `src/middleware/auth.js`)
  Target: API auth guard function (e.g., `authRequired(req,res,next)`)
  Change: Implement/ensure logic: if no authenticated session/user context, respond with **401** for `/api/*` requests (JSON-safe), otherwise `next()`.

* Target File: **App bootstrap** (MISSING: `app.js` / `server.js`)
  Target: middleware/route mounting order
  Change: Ensure session/cookie parsing + auth initialization runs **before** mounting `/api` routes, so the auth middleware can reliably detect unauthenticated requests.

### 4.3 RootCause

RootCause:

* The `POST /api/courses/:course_id/assignments` endpoint is either **not protected by auth** or does not **check the caller’s course role**, allowing a logged-in student to create assignments (returns `200/201`).
* Role enforcement is missing/incorrect at the **route/controller level**: the handler should verify the requester is a course member with an allowed role (e.g., `teacher/admin/assistant`) before creating the assignment.
* The membership role source of truth (membership table/repo) is not consulted (or consulted with the wrong user id / wrong course id), so the role check is effectively bypassed.

---

### 4.4 ActionablePlan

ActionablePlan:

* Target File: **API router for course assignments** (MISSING)
  Target: `POST /api/courses/:course_id/assignments`
  Change: Add authentication guard first; if no authenticated user context, return **401**.

* Target File: **Assignments controller** (MISSING)
  Target: assignment creation handler (e.g., `createAssignment`)
  Change: Before inserting the assignment, load requester’s membership for `course_id`; if requester is not a member return **403** (or 401 if you treat “not a member” as unauthorized), and if role is `student`, return **403**.

* Target File: **Membership repository/model** (MISSING)
  Target: function used by controllers to fetch membership (e.g., `getMembership(course_id, user_id)` / `getRoleInCourse`)
  Change: Ensure it returns a canonical role string exactly matching stored values (e.g., `'student'`, `'teacher'`, `'admin'`, `'assistant'`) and that it’s used by the assignments controller for authorization.

* Target File: **Auth/session middleware** (MISSING)
  Target: middleware that sets `req.user` / `req.user_id`
  Change: Ensure downstream controllers can reliably access the authenticated user id (so the role lookup uses the right principal).

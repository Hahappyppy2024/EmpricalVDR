### 4.3 RootCause

RootCause:

* `GET /api/courses/:course_id/posts` does not enforce **course membership** for the requesting user (missing course-isolation check), so a user from course2 can read course1 posts (IDOR).
* The membership check is either absent or implemented incorrectly (e.g., it checks “user exists” or “logged in” only, but not “member of that course_id”).
* Enforcement is missing at the correct boundary: it must happen **before** the posts query/response, ideally as a reusable middleware applied to all `/api/courses/:course_id/*` resources (or at least in the posts controller).

---

### 4.4 ActionablePlan

ActionablePlan:

* Target File: **Course posts routes file** (MISSING)
  Target: `GET /api/courses/:course_id/posts`
  Change: Require authentication + course membership guard before calling the posts handler; if unauthenticated return **401**, if authenticated but not a member return **403**.

* Target File: **Posts controller** (MISSING)
  Target: posts list handler (e.g., `listPostsByCourse`)
  Change: Add a pre-check (if not using middleware) that validates `(req.user_id, req.params.course_id)` membership; on failure return **403** and do not query/return posts.

* Target File: **Membership repository/model** (MISSING)
  Target: membership lookup (e.g., `isMember(course_id, user_id)` / `getMembership`)
  Change: Implement or fix query to ensure it checks the membership table for exact `course_id` + `user_id` match; return boolean or role.

* Target File: **Shared course authorization middleware** (MISSING)
  Target: middleware (e.g., `requireCourseMember`)
  Change: Centralize the membership check and apply it to all course-scoped resource routes (at minimum, posts) so course isolation is consistent and not duplicated.
